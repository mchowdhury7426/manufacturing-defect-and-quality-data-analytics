✦Overview

When I joined the factory, I quickly realized that real manufacturing data behaves very differently from academic or consultancy datasets. Data existed, but it was not structured for decision-making: inconsistent formats, manual entry errors, and disconnected sources made even a simple request—like “top defects”—hard to deliver reliably.

I built a complete quality analytics system by turning messy inspection data into an analysis-ready model in Power BI. The system evolved step-by-step: cleaning and standardizing raw data, reshaping it for reliable defect ranking, unifying multiple inspection stages, designing a scalable folder-based database approach, integrating buyer final inspection reports, expanding the model upstream (cutting/recutting/fabric), and finally closing the loop by mapping end-customer feedback (VoG) into operational defects for fast investigation and action.

✓What I built (simple explanation)

  • A clean data foundation from inconsistent Excel/database exports—so results are trustworthy.
  • A defect-ranking Power BI model that works correctly by line, floor, buyer, product, style, PID, and date.
  • A central inspection database combining Inline + Endline + Prefinal to compare stages and find patterns.
  • A scalable folder-based ingestion so the system grows without Excel size limits.
  • Buyer final integration (Power Query + DAX) so final data becomes usable and comparable.
  • Upstream integration (cutting, recutting, fabric) via reference tables for stable keys.
  • VoG integration to translate customer feedback into defect language and connect it to factory actions.
Visual proof: this is where the system becomes operational—teams slice by unit/line/floor/style/buyer/time to locate the exact action zone.

⚙Case Study — Detailed Actions (with interpretation)

Each section explains the real problem, what I did, and why that choice mattered inside a working factory environment.

Challenge 1 — Data sourcing and cleaning

Reality: Data existed but wasn’t analysis-ready. Manual + automatic entry created typos, mixed date formats, buyer name mismatches, and inconsistent IDs across huge volumes.

What I did: Standardized dates, corrected repeated errors systematically, validated fields against official lists, and kept the operator workflow unchanged (transform inside Power BI, not at source).

Why it mattered: This step built trust. Wrong data leads to wrong defect ranking, wrong priorities, and wasted corrective action.

Example correction logic I used in Excel (pattern-based correction + lookup): =IF(COUNTIF(Correction!$A:$A, M2), VLOOKUP(M2, Correction!$A:$B, 2, FALSE), M2)

Challenge 2 — Data structure in Power BI

Reality: Defect names were columns, quantities were rows, and there was no unique record key—so “Top Defects” cannot be calculated reliably with filters.

What I did: Created a unique ID (index + key fields), duplicated the table, unpivoted defect columns into rows (Defect Name + Qty), and connected back with one-to-many relationships.

Why it mattered: This is the foundation of correct analytics. Without it, defect ranking breaks when slicing by buyer/line/date/product.

Challenge 3 — Putting inspection stages together (Inline + Endline + Prefinal)

Reality: Stages had different formats, but quality decisions require comparisons across gates to learn where defects are generated.

What I did: Standardized structures, added inspection stage identifiers, appended into a central database, and built stage-wise + defect-wise analysis through a central unpivoted table.

Why it mattered: This converts isolated checkpoints into a traceable quality system for correlation and faster RCA.

Challenge 4 — Scaling the database (Excel limitations)

Reality: One-file Excel databases fail at scale, but complex tools overwhelm operators.

What I did: Folder-based ingestion: split into standardized files, connect Power BI to folders, and improve templates with lookups + error checks.

Why it mattered: “Scale without stress”: operators keep Excel usability while Power BI behaves like scalable ingestion.

Challenge 5 — Integrating buyer final inspection data (Final + RCA)

Reality: Buyer final reports are more reliable but structured differently (naming + grouped categories).

What I did: Transformed reports in Power Query, built DAX logic, designed a final model, integrated with factory stages, and handled combined categories through filtering logic.

Why it mattered: Lets investigations start from buyer final findings and trace backward into inline/operations/time—supporting PDCA and 5-Why actions.

Challenge 6 — Integrating cutting, recutting, and fabric

Reality: Cutting/recutting needs consistent keys; fabric is contract-based and not naturally keyed by style.

What I did: Built reference tables (style–PID–item type + fabric contract mapping) and connected stages into one traceable system.

Why it mattered: End-to-end visibility: inspect and act across upstream and downstream instead of blaming only one gate.

Challenge 7 — Product image and similar-style logic

Reality: Images + historical similarity support prediction but can slow reports and have access issues.

What I did: Built image library references and created upcoming-style vs similar-style mapping so upcoming styles inherit context from similar history.

Why it mattered: Predictive readiness: helps teams prevent likely defects before ramp-up.

Challenge 8 — Integrating Voice of Guest (VoG) into defect language

Reality: Stakeholders wanted customer review analysis to identify defects, but manual mapping is impossible at scale.

What I did: Used an AI-assisted workflow to convert feedback into structured defect type/category/name and integrated it into the same Power BI ecosystem.

Why it mattered: Closed loop: customer complaint → defect language → factory investigation path (fabric → cutting → sewing → inspection → delivery).

Additional integration — Cross-check consistency (C&C)

Correlating results across managers, officers, inspectors, and buyer findings adds governance—confirming everyone is aligned on procedure and interpretation.

⧉Power BI Data Model — Architecture, Complexity, and Industrial Use

The Power BI model behind this case study is intentionally industrial-grade. It is not a single flat table. It is a connected system designed to support reliable analysis across multiple inspection stages and decision levels.

1) Core architecture

A central inspection table stores shared context (style/PID, buyer, date, line/floor, MO, etc.). A separate unpivoted defect-fact table stores defect name + quantity and connects via unique record IDs. This is what keeps filtering and ranking correct.

2) Why it looks complex

The model reflects factory reality—multiple sources, multiple keys, and multiple stages. Each table exists because one real operational question needs governance: consistent keys, consistent definitions, and controlled relationships.

3) Industrial use

  • Correct defect ranking at any level (line/floor/buyer/style/date).
  • Cross-stage traceability from inline to final and back to origin.
  • RCA support through drill-down paths to operations/time context.
  • Closed-loop improvement by connecting VoG into defect taxonomy.
  • Governance through correlation/consistency checks.

✧Closing Reflection

This started as a request for “top defects,” but the real work was building an operationally sustainable system: standardizing reality-based data, restructuring it into a reliable model, scaling ingestion, integrating buyer final results, connecting upstream processes, and closing the loop with VoG. The outcome is a single analytics environment designed to support faster, clearer, and more consistent quality decision-making.
